- 運用の基本方針
  - `ops` 系コマンドは本番運用用であり、live fetch 前提とする。GitHub Actions からそのまま実行できる状態を保つこと。
  - `fixtures/` はローカル開発専用であり、運用コマンドの前提にしてはならない。
  - コミット対象の成果物は基本的に `data/` 配下とし、品質確認 JSON も運用成果物として扱う。
  - 更新最適化では、一覧ページを確認した上で差分のない会議の詳細 HTML をむやみに再取得しないこと。
- 会議体追加依頼が来たときの基本手順
  1. 既存の階層・親会議体・命名規則を確認し、追加対象の `council_id` と親子関係を固める
  2. 一覧ページから起こせるなら `hierarchy export` で追加し、無理なら `council.json` 生成ロジックや会議体ルールを追加する
  3. 対象会議体の取得・解析ロジックを `src/shingikai/councils/` に実装または追加修正する
  4. fixture を使ったローカル検証用テストを追加する
  5. `ops add <council_id>` で本番運用想定の出力を確認する
  6. `data/_quality/meeting_gap_issues.json` を確認し、想定外の欠番や過剰取得がないかを見る
  7. README や docs に新しい運用前提や例外があれば反映する
- 品質チェック起点で修正するときの基本手順
  1. `data/_quality/meeting_gap_issues.json` と必要なら `data/_quality/fetch_errors.json` を確認し、対象会議体と症状を特定する
  2. 既存 `data/councils/<council_id>/` と公式ページを見比べ、どの段階で欠落や混入が起きているか切り分ける
  3. 原因に応じて `src/shingikai/councils/` または共通処理を修正する
  4. 再現用テストを追加または更新する
  5. `ops repair <council_id>` で再生成し、品質確認 JSON まで更新する
  6. 必要なら関連会議体や親会議体も再確認する
- 指示ショートカット
  - 「データ品質チェック」審議会データの欠番などを解決する
    1. data/_quality/*.jsonを確認し問題が起きているものを把握
    2. どのような処理の分岐や処理そのものの追加で対応可能かを考える
    3. shingikaiデータの取得・解析コードを修正する
    4. データの更新、qualityファイルの再生成を実施する
    5. 再確認
    - よくある原因
      - 会議でないものを開催記録として処理している
      - さらに深ぼったところにある過去の記録を取得していない
      - 「親ページに混在した内部委員会」
